# Setup Guide — Getting the Solution Running

## Prerequisites

| Requirement | Version | Notes |
|---|---|---|
| Visual Studio | 2013 or later (2019/2022 recommended) | Must include **ASP.NET and web development** workload |
| .NET Framework | 4.5 Developer Pack | Included with Visual Studio |
| SQL Server LocalDB | Any version | Included with Visual Studio; verify with `sqllocaldb info` in a terminal |
| IIS Express | Any | Included with Visual Studio |

> **No NuGet restore needed.** All packages are committed to the `packages/` folder in the repository.

---

## Step 1 — Set Up the Database

The solution uses a SQL Server database named `PillID`. You need to create it in your LocalDB instance before running any project.

1. Open **SQL Server Object Explorer** in Visual Studio (View → SQL Server Object Explorer)
2. Expand **SQL Server → (localdb)\MSSQLLocalDB**
3. Right-click **Databases → Add New Database**
4. Name it exactly: `PillID`

Or use the terminal:
```
sqllocaldb start MSSQLLocalDB
sqlcmd -S "(localdb)\MSSQLLocalDB" -Q "CREATE DATABASE PillID"
```

> If you have a database backup (`.bak`) or SQL script from the original author, restore/run it now to populate the schema and data.

## Step 2 — Update Connection Strings

The connection strings in this solution are scattered across several files. Each one must point to your local database.

**Your connection string:**
```
Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False
```

### 2a. `ClassChung\app.config`
This is the class library's own config. It holds the named connection strings that `Settings.Designer.cs` maps to.

Update **all five** `connectionString` values:
```xml
<connectionStrings>
  <add name="ClassChung.Properties.Settings.PillIDConnectionString"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
  <add name="ClassChung.Properties.Settings.PillIDConnectionString1"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
  <add name="ClassChung.Properties.Settings.PillIDConnectionString_Server"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
  <add name="ClassChung.Properties.Settings.PillIDConnectionString_Local"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
  <add name="ClassChung.Properties.Settings.PillIDConnectionString2"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
</connectionStrings>
```

> **Why all five?** The LINQ-to-SQL context (`DataClasses.dbml`) is configured to use `PillIDConnectionString1` specifically. The others exist from earlier development. Updating all of them avoids confusion.

### 2b. `ClassChung\Properties\Settings.Designer.cs`
This auto-generated file contains **hardcoded fallback values** (`[DefaultSettingValueAttribute]`) baked directly into the compiled DLL. If a host application's config file does not contain the connection string, this fallback is used.

Update the `DefaultSettingValueAttribute` on each of the five properties to your LocalDB string:
```csharp
[global::System.Configuration.DefaultSettingValueAttribute(
    "Data Source=(localdb)\\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False")]
```

> **Important:** This file is auto-generated by Visual Studio. If you open `Settings.settings` in the VS designer and save, it will be regenerated. After regenerating, re-apply this change — or update `Settings.settings` (step 2c) first so the designer generates the correct default.

### 2c. `ClassChung\Properties\Settings.settings`
This is the source file the Visual Studio designer uses to regenerate `Settings.Designer.cs`.

Update the `<Value>` element for each of the five settings:
```xml
<Value Profile="(Default)">Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False</Value>
```

Also update the `<ConnectionString>` inside each `<DesignTimeValue>` block:
```xml
<ConnectionString>Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False</ConnectionString>
```

### 2d. `PillIdentifierAPI\Web.config`
The Web API project currently has **no connection strings** in its `Web.config`. When an ASP.NET application loads `ClassChung.dll`, it reads connection strings from its own `Web.config`, not from `ClassChung.dll.config`. Without this entry, the API falls back to the hardcoded default in `Settings.Designer.cs`.

Add a `<connectionStrings>` section inside `<configuration>`:
```xml
<connectionStrings>
  <add name="ClassChung.Properties.Settings.PillIDConnectionString1"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
</connectionStrings>
```

> Only `PillIDConnectionString1` is needed here — that is the name `DataClasses.dbml` resolves at runtime.

### 2e. `PillIdentifierForm\App.config`
Same problem as the API — the WinForms app has no connection strings, so it falls back to the hardcoded default. Add the same entry:
```xml
<connectionStrings>
  <add name="ClassChung.Properties.Settings.PillIDConnectionString1"
    connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
    providerName="System.Data.SqlClient" />
</connectionStrings>
```

### 2f. `PillIdentifierMVC\Web.config`
Two changes needed:

**1. Update `DefaultConnection`** (used by ASP.NET Identity / Entity Framework for the user login database):
```xml
<add name="DefaultConnection"
  connectionString="Data Source=(localdb)\MSSQLLocalDB;AttachDbFilename=|DataDirectory|\aspnet-PillIdentifierMVC.mdf;Initial Catalog=aspnet-PillIdentifierMVC;Integrated Security=True;Encrypt=False"
  providerName="System.Data.SqlClient" />
```

**2. Add ClassChung connection string** (needed once ClassChung is referenced by this project for the pill identification feature):
```xml
<add name="ClassChung.Properties.Settings.PillIDConnectionString1"
  connectionString="Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=PillID;Integrated Security=True;Encrypt=False"
  providerName="System.Data.SqlClient" />
```

## Step 3 — Open and Build the Solution

1. Open `PillIdentifierAPI.sln` in Visual Studio
2. Build the solution: **Build → Build Solution** (or `Ctrl+Shift+B`)
3. All four projects should build without errors. The `packages/` folder is already in the repository — no NuGet restore is needed.

If you see reference errors, right-click the solution → **Restore NuGet Packages** as a fallback.

---

## Step 4 — Running Each Project

### ClassChung
Not directly runnable — it is a class library (DLL). It is used by the other three projects automatically.

### PillIdentifierAPI (Web API)
- **Run:** Right-click `PillIdentifierAPI` → Set as Startup Project → Press `F5`
- **URL:** `http://localhost:6844/`
- **API Help page:** `http://localhost:6844/Help`
- **Test endpoint:** `http://localhost:6844/api/v1/GetData/GetDSThuoc`
- **Requires:** Database to be set up (Step 1) and `Web.config` connection string (Step 2d)

### PillIdentifierForm (WinForms desktop)
- **Run:** Right-click `PillIdentifierForm` → Set as Startup Project → Press `F5`
- **Entry point:** `Program.cs` → opens `Giaodienchinh` (main window)
- **Requires:** Database to be set up (Step 1) and `App.config` connection string (Step 2e)

### PillIdentifierMVC (Web — user website)
- **Run:** Right-click `PillIdentifierMVC` → Set as Startup Project → Press `F5`
- **URL:** `http://localhost:23762/`
- **Requires:** `Web.config` connection strings (Step 2f); ClassChung reference must be added before the pill search feature is implemented

## Dependency Map

```
ClassChung.dll
    ↑ referenced by
    ├── PillIdentifierAPI   (needs ClassChung connection string in Web.config)
    ├── PillIdentifierForm  (needs ClassChung connection string in App.config)
    └── PillIdentifierMVC   (reference not yet added — needed for pill search feature)

SQL Server LocalDB (PillID database)
    ↑ accessed by
    └── ClassChung (via LINQ-to-SQL DataClassesDataContext)

ASP.NET Identity / Entity Framework (aspnet-PillIdentifierMVC.mdf)
    ↑ accessed by
    └── PillIdentifierMVC (separate from PillID, only for user login)
```
